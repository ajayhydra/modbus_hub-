1. Application Overview

ModbusHub is a cross-platform desktop tool for engineers who need to inspect, debug, and configure Modbus-based industrial devices. It supports both Modbus TCP (Ethernet) and Modbus RTU (serial RS-232/RS-485).

Auto-PollContinuously read registers at a configurable millisecond interval

Multi-Device ManagerManage, connect, and monitor a fleet of devices simultaneously
Network ScannerDiscover Modbus TCP devices across an IP address range
Live DashboardMetric cards and real-time polling graph (Qt 6)
Alarm ManagerThreshold-based alarms with event logging
Structured LoggingFilterable log with CSV/text export
Dark ModeLight and dark macOS-inspired colour schemes (Settings tab)

How to perform a read:

  1. Select the Function Code (e.g. Read Holding Registers 0x03).
  2. Enter the Address (0-based). For device register 40001, enter 0.
  3. Enter the Quantity (number of registers/coils).
  4. Choose display Format: Decimal, Hexadecimal, or Octal.
  5. Click Read. Results appear in the Register Values panel.

How to perform a write:

  1. Select a write Function Code (0x05, 0x06, 0x0F, or 0x10).
  2. Enter the target Address.
  3. Enter Value(s) separated by spaces (e.g. 100 200 300).
  4. Click Write. The slave echoes the request on success.

Auto-Poll:

Tick Auto-Poll and set an interval in milliseconds to repeat the last read continuously. Values are appended to the Register Values panel and streamed to the Dashboard graph.

Set the poll interval to at least twice the expected device response time to avoid timeout errors.

3. Tab 2 Gateway

Acts as a transparent protocol converter between Modbus TCP and Modbus RTU.

ModeDirectionUse Case
RTU → TCPListens for Modbus TCP clients; forwards frames to an RTU slave on serialConnect a legacy RTU device to an Ethernet SCADA system
TCP → RTUAccepts Modbus TCP requests; translates to RTU frames on the serial busControl RTU devices from a TCP master (PLC, HMI)
  • TCP Listen IP — 0.0.0.0 listens on all network interfaces.
  • Port — Standard is 502.
  • COM Port / Baud Rate / Parity — Must exactly match the RTU device settings.

Click Start Gateway. All forwarded frames and errors appear in the Gateway Log pane. Click Stop Gateway to shut down.

Do not use the Master tab and Gateway tab on the same COM port simultaneously. They will conflict.

8. Modbus Protocol — Introduction

Modbus is a serial communication protocol developed by Modicon in 1979 for connecting PLCs. It is the most widely deployed open protocol in industrial automation and is supported by thousands of sensors, actuators, and controllers worldwide.

Master / Slave Architecture

  • Exactly one master (client) exists per bus. Only the master initiates requests.
  • Up to 247 slaves (servers), each with a unique address (1–247).
  • Slaves only reply; they never send unsolicited data.
  • Address 0 is broadcast (no response expected).

9. Modbus Data Model

Every device exposes up to four independent data tables. ModbusHub addresses all tables using 0-based register addresses (the wire address), which is one less than the 1-based notation common in device datasheets.

Table1-based RangeWire AddressTypeMaster AccessTypical Use
Coils00001 – 099990 – 99981-bitR + WDigital outputs (relay, valve)
Discrete Inputs10001 – 199990 – 99981-bitR onlyDigital inputs (switch, sensor)
Input Registers30001 – 399990 – 999816-bit wordR onlyMeasurements (temperature, flow)
Holding Registers40001 – 499990 – 999816-bit wordR + WSet-points, configuration
Datasheet says register 40010? Enter 9 in ModbusHub (40010 − 40001 = 9).

11. Modbus TCP Frame Structure (MBAP)

Modbus TCP encapsulates the PDU inside a 7-byte MBAP header (Modbus Application Protocol header) sent over a standard TCP socket on port 502. There is no CRC because TCP provides its own error checking.

13. Exception (Error) Codes

When a slave cannot execute a request it returns an exception response: the FC byte has its high bit set (FC | 0x80), followed by a one-byte exception code.

CodeNameMeaning & Common Cause
0x01Illegal FunctionFC not implemented by this device
0x02Illegal Data AddressRegister address out of range for this device
0x03Illegal Data ValueValue outside permitted range (e.g. quantity too large)
0x04Server Device FailureUnrecoverable internal error in the slave
0x05AcknowledgeRequest accepted; processing is delayed — poll for completion
0x06Server Device BusySlave busy with a long operation; retry later
0x08Memory Parity ErrorExtended memory read/write failed parity check
0x0AGateway Path UnavailableGateway cannot find route to target device
0x0BGateway Target No ResponseTarget device did not respond within timeout

14. Request / Response Transaction Flow

  1. Master builds the PDU: Function Code + start address + quantity (read) or value(s) (write).
  2. Master wraps the PDU: RTU prepends slave address and appends CRC-16; TCP prepends 7-byte MBAP header.
  3. Master transmits the frame over the serial line or TCP socket.
  4. Slave validates frame: checks its own address and CRC (RTU) or MBAP (TCP).
  5. Slave executes the function (reads register, writes coil, etc.).
  6. Slave builds a response PDU: mirrored FC + byte count + data, or exception FC + exception code.
  7. Slave wraps and sends the response using the same framing rules.
  8. Master receives, validates, and delivers data to the application.
